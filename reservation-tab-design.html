03

레이아웃 — 세 후보와 선택

trade-offs

A. 카드 리스트

▶ 채택
기존 도구버튼+모달에 그대로 안착 — 신규 코드 최소
탑승권 은유가 조회에 직관적, 타입 필터·복사·문서 버튼 얹기 쉬움
예약은 6~8건뿐 — 스크롤 짧고 카드가 딱 맞음
시간축 위 공백(대기)은 안 보임 → 시간순 정렬로 보완

B. 일정 타임라인

보조 채택
이동/체류를 시간축에서 직관적으로 파악
예약 6건에 전용 타임라인은 과설계, 렌더 코드 큼
"확인번호만 급히" 라는 현장 목적과 어긋남(조회 마찰↑)
→ 카드 기본 정렬을 시간순으로 두어 타임라인 이점만 흡수

C. 일정 탭에 통합

미채택
stop에 이미 reservation 필드 존재 → 배지로 표식 가능
70개 맛집 등과 섞여 예약만 훑기가 불가능
일정(지도 중심)과 예약(문서·번호 중심)은 목적이 다름
→ 통합 대신 일정→예약 점프 배지로 얇게 연결

결론: 독립 카드 리스트 모달을 시간순 정렬로 기본 제공(타임라인의 실익 흡수), 타입 칩으로 필터. 일정 탭에는 통합하지 않되 🎫 배지로 얇게 링크. 조회 마찰이 가장 낮은 조합.

조회 중심 UX 원칙

  • 0-탭 도달 — 🎫 버튼 한 번 → 시간순 카드. 오늘·다가오는 예약이 위로.
  • 3초 스캔 — 확인번호·시각·편명이 카드 표면에 볼드/모노. 펼치지 않아도 게이트에서 읽힘.
  • 탭당 하나의 동작 — 카드=상세, 📍=길찾기, 🧾=문서, 복사=번호. 오조작 여지 최소.
  • 오프라인 내성 — 번호·시각·좌석은 로컬 캐시로 지도 없이도 보임(기존 localStorage 캐시 재사용). 지도/PDF만 네트워크 필요.
04

추가 · 수정 · 보기 흐름

mobile-first

기본 진입은 보기(read-only). 편집은 명시적 진입. 경비 탭의 add/list 전환과 submitEntry→upload→doSave 시퀀스를 그대로 따른다.

보기 (기본)

🎫 예약 버튼 탭

모달 열림 → 시간순 카드 리스트(조회 기본 화면).

타입 칩으로 좁히기(선택)

전체 / 항공 / 숙소 / 열차. 상태는 모달 세션 동안만 유지.

카드 탭 → 상세

전체 필드 + 문서 열기. 확인번호 탭 시 마스킹 해제 + 클립보드 복사.

추가 / 수정

＋ 추가 (또는 상세의 ✎ 수정)

같은 모달 안에서 폼으로 전환. 수정은 기존 값 프리필(경비의 startEdit와 동일).

종류 선택 → 필드셋 접힘/펼침

항공/숙소/열차 세그먼트에 따라 노출 입력만 바뀜. 공통 필드는 항상 표시.

문서 첨부(선택)

PDF·이미지 선택 → downscale()(이미지) 또는 원본(PDF) → upload 액션으로 드라이브 저장 → 링크 회수. 영수증 업로드와 동일 경로.

저장

확인번호·영문명 등 민감값은 로컬에만, 비민감 메타는 시트에 add/update. 저장 확인 후 리스트로 복귀 + 토스트.

모바일 우선 세부 — 입력 최소 44px 터치타깃(기존 .ef input{min-height:44px} 계승), inputmode/type=date 활용, 저장 버튼 하단 고정 폭 100%, 카드 스와이프 없이 탭 위주(오조작 방지).
05

기존 도구버튼+모달 패턴과의 통합

reuse map

새 코드는 거의 없다. 대부분 경비/영수증 스택의 검증된 함수를 재사용한다. 아래는 무엇을 재사용(초록)하고 무엇만 새로(주황) 만드는지.

🎫 도구버튼
#tool-btns에 .tool-b 버튼 1개 추가. 스코어·환율·경비·잔액 옆.
재사용: .tool-b · #tool-btns
🗂 모달 셸
.wk-ov > .wk-card > .wk-x 그대로. 내부에 리스트/상세/폼 cur 전환.
재사용: .wk-ov/.wk-card/.wk-x
☁ 데이터
Apps Script list/add/update/delete + JSONP. 시트에 reservations 시트/열 추가만.
재사용: jsonp · postEntry
📄 문서 업로드
이바우처 → upload 액션 → DriveApp 비공개 저장 → 링크. receiptAnchor()로 렌더.
재사용: downscale · upload · receiptAnchor

새로 만드는 것 (최소)

  • 레코드 스키마 + 타입별 필드셋 렌더 신규 — 카드/상세/폼의 타입 분기. 경비의 tabAdd/tabList에 대응하는 resvList/resvDetail/resvForm.
  • 마스킹 + 로컬-민감값 계층 신규 — confirmation·paxNames는 localStorage(wk_resv_secret_v1)에, 시트엔 비민감 메타만. 상세에서 로컬값과 병합.
  • PDF 업로드 허용 소폭 — 기존 downscale는 이미지 전용. PDF는 다운스케일 없이 base64 직송(용량 가드만). Apps Script upload 핸들러는 mimeType 이미 범용.
  • 일정→예약 점프 배지 선택 — stop에 매칭 예약 있으면 🎫 배지, 탭 시 예약 모달을 해당 카드로 연다.
구현 규모 감각. 데이터 왕복·업로드·모달·토스트가 전부 존재하므로, 실질 작업은 ① 카드/상세/폼 3뷰 렌더 + ② 민감값 로컬 계층 + ③ 시트 열 추가. 스택은 이미 프로덕션 검증됨(경비·잔액·영수증).

민감정보 — 무엇을 어디에 두나

privacy
전제: index.html은 GitHub Pages 공개. expense.token이 이미 노출된 사실상 공개 비밀 → 시트도 익명 접근 가능. 따라서 시트에 평문 PNR·여권영문명·바우처 PDF를 두면 URL을 아는 누구나 열람한다. 이것이 설계의 핵심 제약.

데이터민감도저장 위치공개 노출
확인번호(PNR)높음기기 localStorage목록엔 ••••7QF2 마스킹만
탑승자 영문명높음기기 localStorageG** K** 이니셜

편명·시각·구간·좌석낮음공개 시트 OK노출돼도 무해
호텔명·주소·박수낮음공개 시트 OK일정에 이미 공개 중

선택지와 권장

  • 권장 (기본): 로컬-민감값 + 마스킹 + 드라이브 문서. 추가 인증 없이 지금 스택으로 가능. 단, 민감 원문은 입력한 기기에서만 온전히 보임(4명이 각자 자기 예약만 넣거나, 대표 1인이 넣고 필요 시 문서 링크로 공유).
  • 대안 1: 드라이브 문서로 일원화. 카드엔 마스킹만, 원문 전체는 바우처 PDF를 열어 확인. 로컬 계층 없이도 성립 — 가장 단순.
  • 대안 2: PIN 게이트. 예약 탭 진입에 4자리 PIN(일행만 공유). 편의는 낮고, 공개 시트 문제는 못 막으므로 민감값은 여전히 시트 밖이어야 함. 과잉일 가능성.
  • 비권장: 시트 평문 저장. 편하지만 공개 노출. 채택하지 않음.

※ 근본적 격상(비공개 저장소·서버 인증)은 이 앱의 "단일 공개 index.html" 전제를 벗어난다. 현 제약 안의 최선은 민감값을 시트에서 빼고 드라이브+로컬로 분리하는 것.
